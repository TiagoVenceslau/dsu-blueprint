# static exports.DSUOperation

Describe the 4 distinct stages of a DSUOperation on a DSUBlueprint:

  • PREPARATION: handles dependencies for the build process:
  • fromWeb: the the preparation phase retrieves the Seed from the Web, Sets up '@mount' to later perform the mounting operation
  • CREATION: handles the creation of all nested DSUs (typically from the dsu decorator)
  • CLASS: handles the creation of the actual 'end' DSU (typically from the DSU decorator)
  • EDITING: handles the editing of the end DSU and all its available operations:
  • dsuFile
  • fromCache
  • mount

Since these are nested, these run first and their success is mandatory to continue the operation; - Class decorators: DSUBlueprint decorators, responsible for creating the actual DSU from its Blueprint via OpenDSURepository#create - Edit decorators: property decorators that edit the recently created DSU: - dsuFile: writes files; - fromCache: mounts a previously created DSU - fromWeb: mounts a DSU from the ApiHub or WebService - mount: mounts a predefined KeySSI - also FileSystem related ones, addFileFS, etc... - DSU: the DSU object

# static DSUBlueprint()

DSU Blueprint Decorator

Defines a class as a DSU Blueprint, enabling:

  • Automatic CRUD operations, just by updating its corresponding DSUModel instance and running it through the appropriate OpenDSURepository;
  • Automatic validations && easily extendable for added validations;
  • Automatic serialization -> transmission -> deserialization;
  • Controlled accesses: Ability easily to add business logic at key points of any CRUD operations
.
